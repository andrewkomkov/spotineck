One synchronized zone out of every speaker you own.

Spotineck turns a pile of incompatible speakers — HomePod (AirPlay), Google Home / Nest and any Chromecast, plus your host's local audio — into one synchronized zone. Stream to it from Spotify and from any Apple device over AirPlay.

View on GitHub Quick start

HomePod, Alexa/Echo, Google Home and Chromecast live in three incompatible multi-room protocols and don't combine into one group out of the box. Spotineck is the bridge: one input → a synchronized fan-out to every speaker at once.

What it does

🔊

Synchronized zone

AirPlay 2 (HomePod) + Chromecast (Google/TV) + local ALSA, all playing in lockstep.

🎚

Per-speaker delay

Volume and delay (offset) per speaker, right in the UI — fix cross-protocol drift by ear, at runtime, no restart.

🟢

Spotify Connect

The zone shows up as a "spotineck" device in the Spotify app. Pick it and hit play.

﹫

AirPlay receiver

Any audio from Apple — not just Spotify — fans out to the whole group, Chromecast included.

🎛

Spotify Web API

Start music from the UI or an agent without a phone, with rich metadata and catalog search.

🤖

Agent-friendly API

Clean REST + OpenAPI (/docs, /openapi.json) + a WebSocket for real-time state.

📱

Web interface

A polished, Spotify-style, mobile-friendly UI — installable as a PWA.

🔐

AirPlay PIN auth

Authorize PIN-protected AirPlay speakers right from the interface, and rename the device on the fly.

How it works

Spotineck doesn't reinvent the multi-room engine — it uses OwnTone, a synchronized AirPlay 2 / Chromecast player. Each input is a small service that writes PCM into a named pipe; OwnTone reads it as a source and synchronously feeds every enabled output.

ServiceWhat it does
owntoneThe core: reads pipe sources, synchronously fans out to AirPlay 2 / Chromecast / ALSA. Exposes a REST+WS API.
librespotSpotify Connect endpoint → writes to /music/spotify. Device name comes from a file (changed from the UI).
shairportAirPlay 1 receiver → writes to /music/airplay. Publishes mDNS through the single shared avahi.
apiFastAPI: smooths the OwnTone API into clean REST/WS, adds the Spotify Web API, renaming, and serves the web UI.

Quick start

Linux host with Docker + docker compose. Speakers and host on the same network/VLAN.

# clone, set up FIFOs/config/.env, build & run
git clone https://github.com/andrewkomkov/spotineck.git
cd spotineck
make setup
make up
  1. Open http://<host>:8080 — the web interface.
  2. In the Spotify app pick the spotineck device and hit play.
  3. In the Speakers tab, enable the speakers you want — playback starts in sync.
  4. Audio from Apple: pick spotineck in the AirPlay menu.

API for agents & integrations

Clean REST with an OpenAPI schema at /docs and /openapi.json. Quick summary — GET /api/capabilities.

MethodPurpose
GET /api/statefull snapshot: playback + speakers + queue
POST /api/playback/{play,pause,toggle,next,previous}transport
POST /api/playback/volumezone volume
POST /api/speakers/{id}into the group / volume / delay
PUT /api/speakers/groupset the whole group membership at once
GET/POST /api/device-namename in Spotify Connect
WS /wsreal-time state push

Limitations (honestly)

  • A Chromecast receiver is impossible. The Google Cast protocol is closed on the receiver side — there's no open way to "pretend to be a Chromecast" (unlike AirPlay). For Android: Spotify Connect, AirPlay from a Mac, or DLNA/UPnP (planned).
  • TV-Chromecast with a large buffer (often >2 s) is hard to align perfectly — OwnTone caps the offset at ±2000 ms. Small, stable AirPlay↔Cast drift the delay slider removes nicely.
  • Echo/Alexa can't join the synchronized group (no AirPlay/Cast/stream receiver).
